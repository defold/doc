# Defold 문서 한국어 번역 가이드

이 파일은 현재 `docs/ko/manuals`에 있는 한국어 번역을 기준으로 만들었습니다. 기존 번역 파일을 한국어 용어와 문체의 출처로 삼고, 빠른 용어 확인에는 `docs/ko/glossary.txt`를 사용하세요.

## 기본 원칙

설명문은 자연스럽고 명확한 한국어로 번역합니다. Defold에서 사용자가 실제로 보게 되는 이름, API, 파일명, 경로, 메세지 ID, 설정 키, 명령어, 플랫폼명, 서비스명은 원문 그대로 유지합니다.

기존 한국어 번역은 오래된 문서와 최근 문서가 섞여 있어 일부 파일에 영어 원문, 오탈자, 용어 혼용이 남아 있습니다. 이런 흔적은 스타일 규칙으로 삼지 말고, 반복적으로 일관되게 쓰인 용어와 표현을 기준으로 삼으세요.

## 문체와 톤

기존 매뉴얼은 직접적이고 실용적인 설명체를 사용합니다.

- 기본 종결은 `-합니다`, `-됩니다`, `-할 수 있습니다`, `-하세요`를 사용합니다.
- 절차 설명은 `선택합니다`, `추가합니다`, `설정합니다`, `확인합니다`처럼 짧고 명확하게 씁니다.
- 불필요하게 직역하지 말고 한국어 독자가 바로 이해할 수 있는 순서로 문장을 정리합니다.
- 기존 문서에는 `당신`이 자주 나오지만, 새 번역에서는 가능하면 주어를 생략하거나 `사용자`로 바꿔 자연스럽게 씁니다.
- 전문 용어는 첫 언급에서 한국어와 영어를 함께 제시하고, 이후에는 한국어를 사용합니다.

예:

- `컬렉션 프록시(Collection proxy)는 ...`
- `직교 투영(orthographic projection)을 사용하려면 ...`
- `입력 바인딩(input binding)에 매핑되어 ...`

## 번역할 항목

다음 항목은 한국어로 번역합니다.

- 제목, 소제목, brief, 소개 문단
- 개념 설명, 절차 설명, 주의사항, 팁
- 표와 목록의 설명문
- 링크 텍스트가 일반 설명일 때의 링크 텍스트
- 코드 주석 중 API나 식별자가 아닌 설명문
- UI 라벨 뒤에 붙는 설명문

선호 용어:

- `game object` -> `게임 오브젝트`
- `component` -> `컴포넌트`
- `collection` -> `컬렉션`
- `message passing` -> `메세지 전달`
- `project settings` -> `프로젝트 설정`
- `script properties` -> `스크립트 프로퍼티`
- `application lifecycle` -> `어플리케이션 라이프사이클`
- `rendering pipeline` -> `렌더링 파이프라인`
- `input binding` -> `입력 바인딩`

## 원문을 유지할 항목

다음 항목은 영어 또는 원래 표기를 유지합니다.

- 제품, 플랫폼, 기술명: `Defold`, `Lua`, `Android`, `iOS`, `HTML5`, `macOS`, `Windows`, `Linux`, `OpenGL`, `WebGL`
- 파일명, 확장자, 경로: `game.project`, `.apk`, `.ipa`, `.material`, `main.collection`, `/builtins/render/default.renderc`
- API, 함수, 콜백: `msg.post()`, `msg.url()`, `go.get_id()`, `factory.create()`, `init()`, `update()`, `on_message()`
- 메세지 ID와 소켓: `proxy_loaded`, `acquire_input_focus`, `set_time_step`, `@render`, `@system`, `@physics`
- 설정 키와 enum 값: `main_collection`, `include_dirs`, `CONSTANT_TYPE_WORLD`, `FILTER_MODE_LINEAR`
- UI 라벨과 메뉴 경로: `Outline`, `Properties`, `Assets pane`, `File ▸ New`, `Project ▸ Bundle`
- 명령어와 콘솔 출력: `adb install`, `java -jar bob.jar`, `ERROR:GAMEOBJECT`
- 외부 서비스명: `Google Play`, `App Store`, `Amazon S3`, `Play Console`, `GitHub`

UI 동작을 새로 작성할 때는 라벨을 `<kbd>...</kbd>`로 감싸도 됩니다.

- `<kbd>Project ▸ Bundle ▸ Android Application...</kbd>`
- `<kbd>Open With ▸ Text Editor</kbd>`
- `<kbd>Add Component</kbd>`

기존 문서가 `**Project > Build And Launch**`처럼 굵게 표시한 경우, 해당 문서를 전면 수정하는 작업이 아니라면 형식만 바꾸기 위해 손대지 않아도 됩니다.

## Defold 공식 개념

Defold 고유 개념은 첫 언급에서 한국어 용어와 영어 원어를 함께 보여주는 방식을 선호합니다.

- `게임 오브젝트(game object)`
- `컬렉션(collection)`
- `컴포넌트(component)`
- `컬렉션 프록시(Collection proxy)`
- `컬렉션 팩토리(Collection factory)`
- `설계도(blueprint)` 또는 `프로토타입(prototype)`
- `입력 바인딩(input binding)`
- `물리 월드(physics world)`
- `직교 투영(orthographic projection)`

이후 같은 문서 안에서는 한국어 용어만 사용해도 됩니다. 단, 에디터 타입명, 파일 확장자, 속성명, 메세지 ID, API 식별자를 설명할 때는 원문을 유지합니다.

## 주요 표기 선택

기존 번역에서 가장 많이 쓰인 표기를 기준으로 새 번역도 맞춥니다.

- `게임 오브젝트`를 기본으로 사용합니다. 기존의 `게임오브젝트`는 수정 범위에 들어올 때만 정리합니다.
- `메세지`를 사용합니다. 기존 문서에는 `메시지`도 조금 있지만 `메세지`가 압도적으로 많습니다.
- `매뉴얼`을 사용합니다. 기존의 `메뉴얼`은 오탈자로 보고 새 번역에서는 쓰지 않습니다.
- `어플리케이션`, `컨텐츠`, `텍스쳐`, `메터리얼`, `쉐이더`, `프로퍼티`, `해쉬`, `메니페스트`, `포멧`을 사용합니다.
- `id`는 코드와 설정 문맥에서는 그대로 쓰고, 설명문에서는 `아이디` 또는 `식별자`를 사용할 수 있습니다.
- Defold 입력 시스템에서는 기존 한국어 문서의 주된 표기를 따라 `입력 바인딩`, `입력 동작`, `입력 포커스`, `인풋 스택`을 기본으로 사용합니다. 이미 번역된 문서의 문맥을 맞출 때는 `인풋 바인딩`, `입력 액션`, `인풋 액션`처럼 남아 있는 표기도 함께 고려합니다.
- `속성`은 일반 의미에 사용하고, 에디터/API의 property 개념은 `프로퍼티`를 사용합니다.

## 에디터 UI

기존 한국어 문서는 에디터 UI 라벨 대부분을 영어로 유지합니다. 새 번역에서도 임의로 라벨을 번역하지 마세요.

유지할 예:

- `Outline`
- `Properties`
- `Assets pane`
- `Scene Editor`
- `Material Editor`
- `Curve Editor`
- `Add Game Object`
- `Add Component`
- `Add Component File` (현재 영어 문서와 튜토리얼에서 쓰이는 경우)
- `Add Component From File` (기존 한국어 문서나 일부 튜토리얼에 남은 경우)
- `Project ▸ Build And Launch`
- `Project ▸ Bundle`
- `Open With ▸ Text Editor`

필요하면 한국어 설명을 붙입니다.

- `Outline 창`
- `Properties 창`
- `Assets pane 창`
- `Scene Editor 도구`

메뉴 경로를 새로 쓸 때는 `▸`를 사용합니다. 기존의 `>`는 문서를 전면 정리하는 경우가 아니면 그대로 둘 수 있습니다.

## 코드, Markdown, 파일 구조

- YAML front matter를 보존합니다.
- Markdown 제목 계층과 목록 구조를 보존합니다.
- 포함 지시문, 링크 대상, 이미지 경로, 앵커를 변경하지 않습니다.
- 코드 블록의 API, 식별자, 문자열, 파일명, 경로를 번역하지 않습니다.
- 설명용 코드 주석은 한국어로 번역할 수 있지만, 주석 안의 API와 메세지 ID는 유지합니다.
- 기술 리터럴에는 백틱을 사용합니다: `game.project`, `main:/level/hero#brain`, `hash("jump")`.
- UI 액션에는 `<kbd>...</kbd>`를 사용할 수 있습니다.
- 파일명, 확장자, 리소스 id, 예제 오브젝트 id를 번역하거나 이름을 바꾸지 않습니다.

## 제목과 링크

기존 한국어 문서에는 영어 제목이 많이 남아 있지만, 새 번역에서는 일반 제목을 한국어로 번역하는 것이 좋습니다.

- `Message passing` -> `메세지 전달`
- `Application lifecycle` -> `어플리케이션 라이프사이클`
- `Project settings` -> `프로젝트 설정`
- `Building blocks` -> `빌딩 블록`
- `Collection Proxies` -> `컬렉션 프록시`

단, API 이름, UI 라벨, 파일명, 서비스명, 공식 제품명으로 쓰이는 링크 텍스트는 원문을 유지합니다.

## 플랫폼과 외부 서비스

플랫폼명과 서비스명은 번역하지 않습니다.

- `Android`, `iOS`, `HTML5`, `macOS`, `Windows`, `Linux`
- `Google Play`, `App Store`, `Apple Developer`, `Play Console`
- `AWS`, `Amazon S3`, `IAM`, `CORS`, `IndexedDB`, `Emscripten`
- `OpenGL`, `OpenGL ES`, `WebGL`

설명이 필요하면 첫 언급에서 한국어 설명을 덧붙입니다.

- `아마존 웹 서비스(AWS)`
- `버켓(bucket)`
- `개발용 엔진(development engine)`

## 오탈자와 미완성 번역 처리

기존 파일에 남아 있는 오탈자나 오래된 표기는 그대로 따르지 않습니다. 아래 항목은 기존 말뭉치에서 가장 많이 쓰인 표기가 아니라, 새 번역에서 의도적으로 표준화할 교정 표기입니다.

새 번역 또는 문서 정리 중에는 다음을 교정합니다.

- `메뉴얼` -> `매뉴얼`
- `어플레케이션`, `어블리케이션` -> `어플리케이션`
- `엑세스` -> `액세스`
- `플렛폼` -> `플랫폼`
- `Javscript` -> `JavaScript`
- `LUA` -> `Lua`

영어 문장이 통째로 남아 있거나 제목만 영어로 남아 있는 부분은 원문 유지가 필요한 리터럴이 아니라면 한국어로 번역합니다.

## 실무 판단 기준

사용자가 Defold 에디터, API, 파일 시스템, 콘솔, 설정 파일에서 그대로 확인해야 하는 문자열은 원문 그대로 둡니다.

그 문자열이 무엇을 의미하는지 설명하거나 사용 방법을 안내하는 문장은 한국어로 번역합니다.
